手配 / TEHAI — 使い方マニュアル

tehai を使う

要求を「契約付きの小タスクDAG」に分解し、適切なモデル・effort・チームに配って、検証・差し戻し・監査する委譲基盤。Python 標準ライブラリのみ・依存ゼロ。

① はじめに ② 5分で試す ③ コマンド一覧 ④ backend と sandbox ⑤ Multi-Team ⑥ ライブラリ ⑦ 自己改善 ⑧ 検証・テスト ⑨ 安全の約束

1 はじめに

tehai は「大きいモデルに全部やらせない」基盤です。ユーザー要求を Task Contract 付きの小タスクに分解し、登録制のエージェント／チームに配り、リスクに応じてモデル階層・reasoning effort・レビュー構成を選び、Judge が採否を決め、失敗は原因チームへ差し戻します。

準備

  • Python 3.10 以上（このマシンでは python3。python ではない）
  • インストール不要：標準ライブラリのみ・追加依存ゼロ
  • リポジトリへ移動してから実行：
cd ~/Projects/tehai
既定は完全オフラインの決定的モード（実モデル呼び出し無し）。本物の生成・レビューをしたい時だけ --backend を付けます（④参照）。

2 5分で試す

# ① 要求を「契約付きタスクDAG」に分解（オフライン・即時）
python3 -m tehai plan "ログイン画面の入力バリデーションを実装する"

# ② 生成→レビュー→Judge を実行（オフラインはスタブ生成）
python3 -m tehai run "add a pagination helper" --out ./out

# ③ マルチチームで丸ごと（企画→設計→実装→検証→統合→文書）
python3 -m tehai meta "Todoアプリにタスク完了フラグ機能を実装する"

# ④ 登録されているチーム／エージェント／組織テンプレを見る
python3 -m tehai teams
python3 -m tehai agents

これだけでも「どう分解され・誰に割り当てられ・どのモデル/effort で・どうレビューされるか」が全部表示されます。

3 コマンド一覧

コマンド何をする
tehai plan "<要求>"要求を Task Contract 付きの DAG に分解。担当エージェント・モデル階層・effort・レビュー計画を表示。--json で機械可読出力。
tehai run "<要求>"計画した各タスクの成果物を生成 → 成果物をレビュー → Judge → FSM（accept/revise-retry/discard/human-gate）。--out DIR で成果物保存。
tehai review "<要求>"リスク別レビュー計画を実行し Judge が採否（accept/revise/discard/rerun）。--artifact FILE で実物を渡せる。
tehai meta "<目標>"Multi-Team。Meta Orchestrator が必要チームを選び、Team Contract で配分・統合・差し戻し（⑤）。
tehai teams / agents / orgs登録済みのチーム／エージェントテンプレ／組織テンプレを一覧。
tehai evaluate <ledger.jsonl>実行ログから成功率・コスト等の指標と改善提案を集計（提案のみ）。
tehai calibrate <ledger.jsonl>ログから具体的な設定案（ルーター閾値など）を提案。--apply cfg.json で採用可能 config を書き出し（⑦）。
tehai verifyMulti-Team の検証シナリオ（意図的失敗含む）を実行し verification_report.md を生成。
終了コードでCIゲートできます：run は失敗タスクで、review は revise/discard/rerun で、plan はブロック時に非ゼロ終了。

4 backend と sandbox

既定は決定的（オフライン）。実モデルを使うには --backend を指定します。

フラグ意味
--backend null既定。実モデル無し・ヒューリスティック・即時。
--backend ollamaローカル Ollama（このマシンは localhost:11434 / gemma4）。--ollama-host URL で上書き可。
--backend claude-cliclaude CLI 経由（haiku/sonnet/opus を tier で）。
--sandbox生成コードを実際にコンパイル/実行して検証（py_compile・unittest・node --check 等）。OS名前空間でネットワーク遮断・/home,/root を tmpfs で隠す。
--sandbox-strict隔離が使えない環境では実行を拒否（無防備実行を防ぐ）。
# 実モデルで生成し、隔離サンドボックスで実テスト実行
python3 -m tehai run "メール検証関数とその単体テストをPythonで実装" \
    --backend ollama --sandbox --out ./out
注意：マルチチームを実モデルで回すと呼び出し回数が多く遅い（チーム×タスク×レビュー）。高リスク／少数タスクに絞るのが設計思想です。

5 Multi-Team（チーム編成で回す）

Meta Orchestrator が目標から必要チームを選び、Team Contract の DAG で配分します。README系は Documentation だけ、認証系は Security Team が統合前に加わります。

python3 -m tehai meta "ユーザー認証機能を実装する。トークンを安全に生成する"
  • 失敗は failure_type で分類して原因チームへ差し戻し（盲目リトライしない）。例：要件曖昧→企画、統合衝突→統合。
  • Autonomous Loop Guard：低リスクは自律回復、security/反復3回/コスト超過/本番・外部送信は停止＋承認ゲート。
  • 高リスク設計では Cross-Team Competition（保守性/速度/セキュリティ案を比較）。
  • --autonomy supervised|autonomous_low_risk|manual … で自律度、--json で全結果。

6 ライブラリとして使う（Python API）

# 単一チームのパイプライン
from tehai import Orchestrator
plan = Orchestrator.default(backend="null").plan("ログイン検証を実装")
print(plan.to_json())

# マルチチーム
from tehai.teams import MetaOrchestrator
res = MetaOrchestrator.default(backend="null").run("機能Xを実装する")
print(res.final_status, res.metrics)

サンドボックス実行付きは Orchestrator.default("ollama", sandbox=True)。失敗ルーティングの再現は run(goal, injected_failures={"implementation_team": (FailureType.REQUIREMENT_AMBIGUITY, 1)})。

7 自己改善ループ（記録→提案→人間が採用）

# 実行ログを残す → 指標と提案を見る → 採用可能 config を書き出す
python3 -m tehai run "add a helper" --save-log --run-id R-1
python3 -m tehai evaluate runs/R-1.jsonl
python3 -m tehai calibrate runs/R-1.jsonl --apply cfg.json

# 人間が cfg.json を確認してから採用
python3 -m tehai plan "..." --config cfg.json
自己改善は提案まで。本番ロジック（チーム配分・ルーティング）を自動で書き換えることはしません。採用は常に人間の明示的な一手です。

8 検証・テスト

# 全テスト（標準ライブラリ unittest・依存ゼロ）
python3 -m unittest discover -s tests -t .

# Multi-Team 検証シナリオ（意図的失敗含む）→ verification_report.md 生成
python3 -m tehai verify

意図的な失敗（曖昧要件・セキュリティ・統合衝突・反復失敗・コスト超過）を注入して、正しく止まり・正しく差し戻すかを検査します。

9 安全の約束

  • 承認ゲート：本番デプロイ・外部送信・削除・課金・秘密情報・権限変更は人間承認を通す。
  • 最小権限：子の権限は親（チーム）を超えない。レビュアーはコード修正不可、テスターは本番接続不可。
  • 境界：最大再帰深度3・最大委譲20・最大同時8・同一失敗3回で停止。「分解＝進捗」とはみなさない。
  • 監査可能：成果物に provenance、ログは追記専用、決定的で再現可能。
  • サンドボックス：生成コード実行はネットワーク遮断＋秘密ディレクトリ隠蔽。既定OFF・明示オプトイン。
